# Agent Instructions — OpenExperiments

This file provides context and constraints for AI agents (Claude Code, Codex, Cursor, etc.) working in this repository.

## What This Project Is

OpenExperiments is a Next.js 15 app deployed on Cloudflare Pages. It uses Cloudflare D1 (SQLite) as its database via Drizzle ORM. Every route runs on the **Edge Runtime** — no Node.js APIs are available at runtime.

## Codebase Map

| Path                  | What it does                                             |
| --------------------- | -------------------------------------------------------- |
| `src/db/schema.ts`    | Drizzle schema — source of truth for all tables          |
| `src/lib/types.ts`    | TypeScript interfaces shared across client and server    |
| `src/lib/auth.ts`     | Session resolution from cookies                          |
| `src/lib/api.ts`      | Client-side fetch wrappers                               |
| `src/app/api/`        | All API routes (edge runtime)                            |
| `src/app/`            | Next.js App Router pages                                 |
| `drizzle/migrations/` | SQL migrations generated by drizzle-kit                  |
| `scripts/seed.sql`    | Seed data for local and remote D1                        |
| `wrangler.toml`       | Cloudflare project config (D1 binding, build output dir) |
| `mcp-server/`         | Standalone MCP server — separate package, separate deps  |

## Hard Rules

- **Every non-static page and API route must export `export const runtime = "edge"`** — Cloudflare Pages will reject the build otherwise.
- **No Node.js built-ins** (`fs`, `path`, `crypto` from Node, etc.) — use Web APIs (`crypto.randomUUID()`, `fetch`, etc.).
- **All DB access goes through Drizzle** — never write raw SQL in application code.
- **`getRequestContext()` is how you access D1 and env vars** in API routes — import from `@cloudflare/next-on-pages`.
- **Schema changes require a migration** — edit `src/db/schema.ts`, then run `npx drizzle-kit generate`. Never edit migration files by hand.
- **No `any` types** — TypeScript strict mode is enforced.
- **`null` vs `undefined`** — DB columns return `null`; TypeScript interfaces often use `undefined`. Use `?? undefined` when mapping DB rows to types.

## Database Patterns

```ts
// Getting the DB in an API route
import { getRequestContext } from "@cloudflare/next-on-pages";
import { getDB } from "@/db";

const { env } = getRequestContext();
const db = getDB(env); // passes the D1 binding

// Getting the DB in a server component (uses React cache)
import { getDB } from "@/db";
const db = getDB();
```

Dataset IDs are stable:

- `ds-1` = Reddit ChangeMyView (text / persuasion)
- `ds-2` = LaMem (images / memorability)
- `ds-3` = Twitter Persuasion Pairs (text / persuasion)
- `ds-4` = Visual Attention & Memorability (images / memorability)

## Hypothesis Lifecycle

```
proposed → arena_ranked → data_tested → field_validated
```

The `status` column on `hypotheses` tracks this. Arena voting updates `arenaElo` and `winRate`.

## Environment Variables

At runtime, env vars come from `getRequestContext().env`, not `process.env`. In server components, `getDB()` internally calls `getRequestContext()`. Client components fetch from API routes.

Local dev env is in `.env` (gitignored). See `.env.example` for required keys.

## Wrangler Commands

The local wrangler binary has a path issue — always use:

```bash
npm run wrangler -- <command>
# e.g.
npm run wrangler -- d1 execute openexperiments-db --remote --command="SELECT COUNT(*) FROM hypotheses"
npm run wrangler -- pages deployment list --project-name openexperiments
```

## Deployment

```bash
npm run deploy   # builds with next-on-pages, deploys to Cloudflare Pages
```

Migrations must be applied separately — they are not part of the deploy script:

```bash
npm run wrangler -- d1 migrations apply openexperiments-db --remote
```

## Pre-commit Hooks

Husky + lint-staged runs ESLint and Prettier on every commit. If the commit fails, fix the lint errors — do not use `--no-verify`.

## MCP Server

`mcp-server/` is a separate package with its own `package.json` and `node_modules`. Changes there require `cd mcp-server && npm run build`. It calls the app's `/api/*` routes over HTTP — it does not import from `src/`.
